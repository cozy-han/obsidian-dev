# 포인트 적립 배치 - 방향성 검토 문서

## 1. 현황 요약

### 데이터 소스
| 구분 | 테이블/소스 | 단위 | 주요 정보 |
|------|------------|------|----------|
| Revenue Report | `PRD_REVENUE_REPORT_TXN` | 여정(편명) x 탑승객 | PNR(`TRACKING_NUMBER`), 편명(`FLIGHT_NUMBER`), 탑승상태(`STATUS`), `DOCUMENT_AMOUNT` |
| 예약 정보 | `ORD_BOOK_BAS` + `ORD_SEGMNT_BAS` + `ORD_PSNGR_CHRG_TXN` | 예약 단위 | PNR, 여정, 탑승객별 운임(`FARE_AMNT`) |
| 적립예정 포인트(SA) | `PYM_POINT_BAS` / `PYM_POINT_LOG_HIST` | 주문(`ORD_NO`) 단위 | 예약 시 산출된 예상 포인트 금액 |

### 적립예정(SA) 포인트 생성 시점
- **예약 완료 시** (IBE `createBooking`):
  - 전체 FARE 합산: `totalFareAmnt += payDetails.getAmount()` (모든 여정의 FARE 합)
  - SA 적립률 조회: `PYM_POINT_CD.POINT_CREAT_RATE` (SA 코드 기준)
  - SA 포인트 = `totalFareAmnt * (적립률 / 100)`
  - `pointAmount`로 직접 전달하여 `PYM_POINT_BAS`에 INSERT

### 현재 실적립(AC) 처리 방식
- batch `PointSaveJob` → `selectPointRq(pnrNo, airlineCd, flightNo)`
- `ORD_PSNGR_CHRG_TXN.FARE_AMNT` SUM → `totAmnt`로 pay 서비스 전달
- pay에서 `totAmnt * (AC 적립률 / 100)` 으로 적립금 계산

### SA 롤백(SC) 현재 방식
- `MultiAddPoint()` 실행 시 자동으로 도착일+3일 경과한 SA 포인트를 SC(전액 취소) 처리
- **SA ↔ AC 금액 비교/차액 처리 로직은 현재 없음**

## 2. 적립 기준점 비교: Revenue 기준 vs 예약 기준

### 방안 A: Revenue 기준으로 적립

**흐름**: Revenue(TKT+USED) 레코드 → PNR+편명으로 그룹핑 → 예약 테이블에서 금액 조회 → 적립

**장점**
- 실제 탑승 확인된 건만 처리 (탑승 미확인 건 자동 제외)
- Revenue 데이터가 곧 적립 트리거 → 로직이 단순

**이슈**
- Revenue 데이터 적재가 선행 조건 (RevenueReportJob 의존)
- Revenue에 없는 예약 건(Revenue 누락)은 영원히 적립 불가
- Revenue 적재 지연 시 적립도 지연
- Revenue는 **여정(편명) 단위** → 왕복의 경우 편도 탑승만으로도 레코드 존재 → 전체 여정 완료 전에 부분 적립 발생 가능
- Revenue 테이블에 항공사 코드 미저장 → 하드코딩("XU") 필요

### 방안 B: 예약 기준으로 적립

**흐름**: 출발일 경과한 예약 조회 → Revenue에서 탑승 확인 → 탑승 확인된 탑승객/여정만 적립

**장점**
- 예약 데이터가 기준이므로 누락 위험 낮음
- 예약 단위로 SA와의 비교가 자연스러움 (같은 `ORD_NO` 기준)
- 부분 탑승 시 SA 차액 계산이 용이

**이슈**
- Revenue 데이터와 예약 데이터의 매칭 로직 필요 (PNR + 편명으로 JOIN)
- Revenue에 탑승 기록이 없는 구간을 "미탑승"으로 간주하는 기준 필요
- 쿼리 복잡도 증가
- Revenue 적재 타이밍에 따라 탑승 확인이 불완전할 수 있음

### 비교 요약

| 항목 | A. Revenue 기준 | B. 예약 기준 |
|------|----------------|-------------|
| 트리거 | Revenue 레코드 존재 | 출발일 경과 예약 |
| 탑승 확인 | Revenue 자체가 증거 | Revenue와 JOIN 필요 |
| 누락 위험 | Revenue 누락 시 적립 불가 | 예약은 있으나 Revenue 미매칭 시 판단 기준 필요 |
| SA 비교 | ORD_NO 매핑 추가 작업 필요 | ORD_NO 기준으로 자연스러움 |
| 부분 탑승 처리 | 탑승한 여정만 처리 (자연스러움) | 미탑승 여정 식별 후 차감 |
| 구현 난이도 | 낮음 | 중간 |

## 3. 적립 단위 비교: 여정 단위 vs 티켓(전체 여정 완료) 단위

### 방안 1: 여정(구간) 단위 적립

탑승이 확인된 **각 여정(편명)별**로 해당 구간의 운임 기준 포인트 적립

**예시 (서울↔제주 왕복, 총 운임 200,000원)**
```
- 가는편 탑승 확인 → 100,000원 기준 적립 (1,000P at 1%)
- 오는편 탑승 확인 → 100,000원 기준 적립 (1,000P at 1%)
- 총 적립: 2,000P
```

**장점**
- Revenue 레코드 단위와 자연스럽게 매핑
- 편도만 탑승해도 해당 구간 포인트 즉시 적립
- 구현이 단순 (현재 설계와 유사)

**이슈**
- SA(적립예정)는 **전체 여정 합산 금액** 기준으로 산출 → 여정 단위 적립과 단위 불일치
  - SA 200,000원 기준 2,000P 예상 → 가는편 적립 시 SA를 어떻게 처리?
  - SA 전액 롤백 후 구간별 AC? → SA 롤백 타이밍 문제
  - SA 부분 롤백? → 구간별 SA 비율 계산 필요
- `ORD_PSNGR_CHRG_TXN.FARE_AMNT`가 여정별로 분리 저장되는지 확인 필요
  - 만약 **전체 합산 금액만 저장**된다면 구간별 운임 분리가 불가능
- Revenue의 `DOCUMENT_AMOUNT`와 예약 테이블의 `FARE_AMNT` 불일치 가능성

### 방안 2: 티켓(전체 여정 완료) 단위 적립

**모든 여정의 탑승이 확인**된 후 전체 운임 기준으로 한 번에 적립

**예시 (서울↔제주 왕복, 총 운임 200,000원)**
```
- 가는편 탑승 확인 → 적립 보류
- 오는편 탑승 확인 → 전체 200,000원 기준 적립 (2,000P at 1%)
```

**장점**
- SA와 동일한 단위 (전체 여정 합산 금액) → SA ↔ AC 비교가 간단
- 기존 `selectPointRq` 쿼리가 이미 PNR 단위 합산 → 그대로 활용 가능
- 적립 금액 = SA 예상 금액 (변동 없으면)

**이슈**
- "모든 여정 탑승 완료" 판정 기준이 필요
  - Revenue에 해당 PNR의 모든 여정(=예약의 모든 SEGMNT)이 TKT+USED로 존재하는지 확인
  - 예약 변경(구간 추가/삭제)이 있었으면 SEGMNT 수 불일치 가능
- 편도만 탑승하고 나머지를 no-show한 경우 → 적립 시점이 불명확
  - 오는편 출발일+N일까지 Revenue가 없으면 미탑승 확정?
  - 확정 시 탑승한 구간만 적립? 아니면 전체 미적립?
- Revenue 적재 지연으로 일부 여정이 늦게 들어오면 판정이 달라질 수 있음

### 비교 요약

| 항목 | 1. 여정 단위 | 2. 티켓(전체 여정) 단위 |
|------|-------------|----------------------|
| 적립 시점 | 각 여정 탑승 확인 즉시 | 모든 여정 탑승 확인 후 |
| SA 비교 | 단위 불일치 (SA=전체, AC=구간) | 단위 일치 (SA=AC=전체) |
| 부분 탑승 | 탑승한 구간만 적립 (명확) | 판정 기준 필요 (N일 대기?) |
| 운임 분리 | 구간별 FARE 분리 필요 | 전체 합산 사용 (기존 로직) |
| 구현 복잡도 | SA 처리가 복잡 | 전체 여정 완료 판정이 복잡 |
| 고객 관점 | 탑승 즉시 포인트 확인 가능 | 왕복 완료 후에야 포인트 확인 |

## 4. SA(적립예정) ↔ AC(실적립) 비교 처리

### 현재 상태
- SA: 예약 시 `전체 FARE × SA 적립률`로 산출, `pointAmount`로 직접 저장
- AC: 배치에서 `ORD_PSNGR_CHRG_TXN.FARE_AMNT` 합산 → `totAmnt × AC 적립률`로 산출
- SC(SA 취소): 도착일+3일 후 **SA 전액 롤백** (금액 비교 없이)

### SA와 AC의 적립률이 다를 수 있는가?
- SA는 `PYM_POINT_CD`에서 `SA` 코드의 `POINT_CREAT_RATE` 사용
- AC는 `PYM_POINT_CD`에서 `AC` 코드의 `POINT_CREAT_RATE` 사용
- **현재 SA와 AC의 적립률이 동일한지 확인 필요**

### 금액 차이 발생 시나리오

| 시나리오 | SA 예상 | AC 실제 | 원인 |
|----------|---------|---------|------|
| 정상 왕복 | 2,000P | 2,000P | 차이 없음 |
| 편도만 탑승 | 2,000P | 1,000P | 오는편 no-show |
| 예약 변경 (업그레이드) | 2,000P | 2,500P | 운임 증가 |
| 부분 취소 후 탑승 | 2,000P → SC 1,000P → SA 잔액 1,000P | 1,000P | IBE에서 이미 부분 SC 처리 |
| 적립률 변경 | 2,000P (5%) | 1,600P (4%) | 적립률 정책 변경 |

### 처리 방안 (논의 필요)

**A안: 단순 교체 (현재 방식 유지)**
- SA 전액 SC 처리 → AC 새로 적립
- SA와 AC 간 금액 비교 없음
- 장점: 단순, 이미 구현됨
- 단점: 고객에게 포인트 감소 → 증가 로 보임

**B안: 차액 정산**
- AC 산출 후 SA와 비교
- AC > SA: 차액만큼 추가 적립
- AC < SA: 차액만큼 SA 부분 롤백
- AC = SA: SA를 AC로 전환 (코드만 변경)
- 장점: 고객 경험 자연스러움
- 단점: 전환 로직 구현 필요

## 5. 확인/결정 필요 사항

1. **적립 기준점**: Revenue 기준(A) vs 예약 기준(B)?
2. **적립 단위**: 여정 단위(1) vs 티켓 단위(2)?
3. **SA ↔ AC 처리**: 단순 교체(A안) vs 차액 정산(B안)?
4. **`ORD_PSNGR_CHRG_TXN.FARE_AMNT`**: 여정별 분리 저장인지, 전체 합산 저장인지?
5. **SA/AC 적립률**: 동일한지, 다를 수 있는지?
6. **부분 탑승 정책**: 편도만 탑승 시 해당 구간만 적립? 미적립? 대기?
7. **미탑승 확정 기준**: 출발일 + N일 후 Revenue 미존재 시 미탑승 확정?
