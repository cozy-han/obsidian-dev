## 1. 전체 구동 개요

```
┌─────────────────────────────────────────────────────────────────┐
│  Bitcoin Auto Trader                                            │
│                                                                 │
│  1. 시작 → DB 초기화 → 마켓 동기화                                 │
│  2. 스케줄러 → 캔들 수집 (일봉/시간봉) + 마켓 선택                  │
│  3. WebSocket → 실시간 시세 수신 → 리스크 모니터링                   │
│  4. 전략 신호 → 리스크 체크 → 주문 실행 → 포지션 관리                │
│  5. 모든 이벤트 → Outbox 보장 → 모니터링 → Discord 알림 + DB 이력   │
│  6. 백테스팅 → 전략 시뮬레이션 → 결과 시각화                        │
│                                                                 │
│  ※ 현재 상태: 전략 자동 평가(4번의 시작점)는 미구현                   │
│     데이터 수집/리스크/주문/알림은 모두 동작                         │
└─────────────────────────────────────────────────────────────────┘
```

### 모듈 의존성 구조

```
           core (공유 커널: 도메인, 이벤트, 포트)
          ↗  ↑   ↑      ↑
api-client   │   │  monitoring
       ↗     │   │
 strategy    │   │
       ↗     │   │
 execution ──┘   │
                 │
     app ────────┘ (모든 모듈 조립 + Outbox/Scheduler)
```

- **core**: 도메인 모델, 이벤트, 포트 인터페이스 (의존성 없음)
- **api-client**: core 의존. Upbit REST/WebSocket 통신
- **strategy**: core + api-client 의존. 전략 엔진, 백테스팅
- **execution**: core + api-client 의존. 주문 실행, 리스크 관리
- **monitoring**: core 의존. Discord 알림, 이벤트 리스너
- **app**: 전체 모듈 조립. Outbox, Scheduler, 메인 클래스

## 2. 시작 시퀀스 (Startup)

앱이 시작되면 다음 순서로 초기화됩니다:

```
./gradlew :app:bootRun --args='--spring.profiles.active=dev'
  │
  ├─ 1. Spring Context 초기화
  │     ├─ @SpringBootApplication (컴포넌트 스캔: com.autocoin.*)
  │     ├─ Virtual Thread 활성화 (spring.threads.virtual.enabled=true)
  │     └─ 전체 모듈의 Bean 등록 (core/api-client/strategy/execution/monitoring/app)
  │
  ├─ 2. Tomcat 시작 (port 8000)
  │
  ├─ 3. DB 연결 (HikariCP → PostgreSQL)
  │     ├─ pool: max 10, min idle 5
  │     └─ timeout: 30초
  │
  ├─ 4. Flyway 마이그레이션 (V1 → V2 → V3)
  │     ├─ V1: 핵심 테이블 10개
  │     ├─ V2: Outbox 패턴 (outbox_events)
  │     └─ V3: 알림 이력 (alert_history)
  │
  ├─ 5. JPA 스키마 검증 (ddl-auto: validate)
  │
  ├─ 6. Bean 생성 (인프라 초기화)
  │     ├─ [api-client] UpbitClient (RestClient + Resilience4j)
  │     │   ├─ RestClient: base URL, connect 5s, read 10s, SimpleClientHttpRequestFactory
  │     │   ├─ quotationRateLimiter: 10회/초 (시세 조회)
  │     │   ├─ exchangeRateLimiter: 30회/초 (계정 API)
  │     │   ├─ orderRateLimiter: 8회/초 (주문)
  │     │   └─ upbitCircuitBreaker: 실패율 50% → OPEN → 1분 대기 → HALF_OPEN(3회)
  │     │
  │     ├─ [api-client] UpbitWebSocketClient (Bean만 생성, 연결은 안 함)
  │     ├─ [api-client] UpbitJwtTokenGenerator (HS512, access_key/secret_key)
  │     ├─ [api-client] Caffeine Cache: marketList(10분 TTL), ticker(5초 TTL)
  │     │
  │     ├─ [strategy] StrategyFactory, StrategyService, BacktestService
  │     ├─ [execution] TradingCoordinator, OrderExecutionService
  │     ├─ [execution] RiskManagementService, PositionManagementService
  │     ├─ [monitoring] DiscordWebhookSender, AlertManagementService
  │     ├─ [monitoring] TradingEventListener, RiskEventListener, SystemEventListener
  │     │
  │     └─ [app] OutboxDomainEventPublisher, OutboxEventProcessor
  │
  ├─ 7. db-scheduler 활성화 → 6개 스케줄 작업 등록
  │     ├─ threads: 6, polling: 10초, heartbeat: 5분
  │     └─ (상세는 섹션 3.1 참조)
  │
  └─ 8. ApplicationReadyEvent 발생
        └─ MarketSyncService.init() (@EventListener)
            │
            ├─ exchangeClient.getMarkets()
            │   └─ UpbitClient → quotationRateLimiter → CircuitBreaker
            │   └─ GET https://api.upbit.com/v1/market/all (인증 불필요)
            │
            ├─ 전체 마켓 응답 (수백 개)
            │   └─ MarketRsMapper.toDomainList() → List<Market>
            │
            ├─ KRW 마켓만 필터링 (market.marketCode.isKrwMarket)
            │
            ├─ 각 마켓에 대해:
            │   ├─ DB에 없으면 → marketRepository.save(newMarket)
            │   └─ DB에 있고 marketWarning 변경 → updateWarning() → save()
            │
            └─ 로그: "Market sync completed: 5 new, 0 updated, 180 total KRW markets"
               ※ 실패 시 경고 로그만 출력, 앱은 정상 실행
```

### 시작 시 주의사항

| 상황 | 결과 |
|------|------|
| API 키 없음 | 앱 정상 시작. 인증 필요 API 호출 시에만 실패 |
| DB 연결 실패 | 앱 시작 실패 (필수) |
| Upbit API 장애 | 마켓 동기화 스킵, 경고 로그 출력 |
| Discord 미설정 | 알림 자동 스킵 (`isAlertsConfigured()` 체크) |

## 3. 상시 동작 (Runtime)

앱이 시작된 후 아래 작업들이 자동으로 실행됩니다.

### 3.1 스케줄러 (db-scheduler, 자동 실행)

| 작업명 | 주기 | 클래스 | 동작 |
|--------|------|--------|------|
| `collect-daily-candles` | 매일 00:05 KST | `CandleCollectionService` | **전체** 마켓 일봉 1개 수집 |
| `collect-hourly-candles` | 매시간 :05 | `CandleCollectionService` | **활성** 마켓 시간봉 1개 수집 |
| `sync-markets` | 매일 09:00 KST | `MarketSyncService` | Upbit 마켓 목록 재동기화 |
| `select-markets` | 매일 00:10 KST | `MarketSelectionService` | 거래대금 상위 20개 종목 선택 |
| `process-outbox-events` | 60초 간격 (FixedDelay) | `OutboxEventProcessor` | PENDING → PROCESSED 마킹 |
| `retry-failed-outbox-events` | 5분 간격 (FixedDelay) | `OutboxEventProcessor` | FAILED 이벤트 재시도 (최대 3회) |

> **일봉 수집 주의**: `collectDailyCandles()`는 `findAll()`로 전체 마켓을 대상으로 수집합니다. 시간봉만 `findByActive(true)`로 활성 마켓만 수집합니다.

#### 마켓 선택 상세 흐름

```
selectMarketsTask (매일 00:10 KST)
  │
  ├─ properties.enabled 확인 (false면 스킵)
  │
  ├─ 최근 7일간 일봉 데이터 조회
  │   candleRepository.findByCandleTypeAndDateRange(DAYS, 7일 전 ~ 현재)
  │
  ├─ 종목별 평균 거래대금 계산 (calculateAverageTradePrice)
  │   candles를 marketCode별 그룹핑 → 그룹별 accTradePrice 평균
  │
  ├─ 필터링: 평균 거래대금 >= 10억원 (minTradePriceKrw)
  │
  ├─ 거래대금 내림차순 정렬 → 상위 20개 (topN)
  │
  ├─ 기존 마켓과 비교하여 활성화/비활성화 변경:
  │   ├─ 새로 선택된 종목 → market.activate() → save
  │   └─ 제외된 종목 → market.deactivate() → save
  │
  └─ 변경 있으면 → MarketSelectionChangedEvent 발행
      (activatedMarkets, deactivatedMarkets, totalActiveCount)
```

#### 캔들 수집 상세 흐름

```
collectForMarkets(markets, candleType, count)
  │
  ├─ 각 마켓에 대해 순차적으로:
  │
  │   exchangeClient.getCandles(marketCode, candleType, count)
  │     └─ UpbitClient → quotationRateLimiter(10/s) → CircuitBreaker
  │     └─ GET /candles/{days|minutes/60}?market={code}&count={count}
  │
  │   CandleRsMapper.toDomainList() → List<Candle>
  │
  │   중복 체크: existsByMarketCodeAndCandleTypeAndCandleDateTimeUtc()
  │     ├─ 이미 존재 → 스킵
  │     └─ 신규 → candleRepository.saveAll(newCandles)
  │              → MarketDataReceivedEvent 발행
  │
  └─ 예외 발생 시 해당 종목만 스킵, 다음 종목으로 계속
```

### 3.2 WebSocket 실시간 데이터

```
subscribeTicker(activeMarketCodes) 호출 시
  │
  ├─ WebSocket 연결: wss://api.upbit.com/websocket/v1
  │
  ├─ 구독 메시지 전송:
  │   [
  │     {"ticket": "uuid"},
  │     {"type": "ticker", "codes": ["KRW-BTC","KRW-ETH",...], "isOnlyRealtime": false}
  │   ]
  │
  └─ 실시간 Ticker 수신 (반복)
       │
       ├─ onMessage(bytes: ByteBuffer)
       │   └─ UTF-8 디코딩 → processMessage()
       │
       ├─ objectMapper.readValue(message, TickerRs.class)
       │   └─ TickerRsMapper.toDomain() → MarketInfo 도메인 객체
       │
       └─ eventPublisher.publishEvent(TickerUpdatedEvent(marketInfo))
            │
            └─ RiskManagementService.onTickerUpdated() 수신
                 ├─ 각 오픈 포지션에 대해:
                 │   ├─ 손절 체크: 수익률 <= -3%?
                 │   ├─ 익절 체크: 수익률 >= +5%?
                 │   └─ 조건 충족 시 RiskBreachedEvent 발행
                 │
                 └─ 전체 하드리밋 체크
                      └─ 전체 손실률 <= -15% → 매매 전면 중지
```

**WebSocket 재연결 전략** (WebSocketReconnectManager):

```
onClose(code, reason, remote)
  │
  ├─ SystemErrorEvent("WebSocket disconnected") 발행 → Discord 알림
  │
  └─ scheduleReconnect()
       │
       ├─ shouldRetry()? (attemptCount < 10)
       │   ├─ NO → SystemErrorEvent("MAX_RETRIES") → 포기
       │   └─ YES → getNextDelay()
       │
       └─ scheduler.schedule(reconnect(), delay)
            ├─ 1회차: 1초 후
            ├─ 2회차: 2초
            ├─ 3회차: 4초
            ├─ ... (지수 백오프, 배수 ×2)
            └─ 최대 300초 (5분)
```

> **참고**: `subscribeTicker()`는 수동 호출이 필요합니다. 앱 시작 시 자동으로 WebSocket에 연결하지 않습니다.

### 3.3 Outbox 이벤트 처리

```
비즈니스 로직 실행 (같은 @Transactional)
  │
  ├─ DomainEventPublisher.publish(event)
  │   │
  │   └─ OutboxDomainEventPublisher.publish()
  │        ├─ ① OutboxEvent 생성 (eventId, eventType, payload=JSON직렬화)
  │        ├─ ② outboxEventRepository.save() → outbox_events 테이블 INSERT (PENDING)
  │        └─ ③ applicationEventPublisher.publishEvent() → Spring 이벤트 즉시 발행
  │
  └─ COMMIT (①②③ 원자적 보장)
```

```
OutboxEventProcessor (60초 간격)
  ├─ findPendingEvents(limit=100) → PENDING 상태 조회
  └─ 각 이벤트: status = PROCESSED, processedAt = now() → save

OutboxEventProcessor (5분 간격)
  ├─ findFailedEvents(maxRetry=3) → FAILED & retryCount < 3 조회
  └─ 각 이벤트:
      ├─ 성공 → status = PROCESSED
      └─ 실패 → retryCount++, lastError 기록
```

> 현재 모놀리스에서는 이벤트가 즉시 in-process로 전달되므로, Outbox는 감사 추적(audit trail) 역할을 합니다. MSA 전환 시 메시지 브로커(Kafka 등) 전송으로 변경됩니다.

## 4. 매매 흐름 (Trading Flow)

### 4.1 전체 흐름도

```
[데이터 수집]          [전략 분석]           [주문 실행]          [포지션 관리]

캔들 수집 ─────┐
(스케줄러)      │
               ├─→ 전략 평가 ─────→ TradingSignalEvent
WebSocket ─────┘   (※ 미구현)        │
(실시간 시세)                         ├─ BUY ──→ 리스크 체크
                                     │          ├─ 승인 → 매수 주문 → 포지션 생성
                                     │          └─ 거부 → 로그
                                     │
                                     ├─ SELL ─→ 포지션 확인
                                     │          └─ 매도 주문 → 포지션 종료
                                     │
                                     └─ HOLD ─→ 무시

[리스크 모니터링]

TickerUpdatedEvent
  └─ 손절/익절/하드리밋 체크
       └─ RiskBreachedEvent → 강제 매도 → 포지션 종료
```

### 4.2 전략 분석 흐름

```
StrategyService.analyzeMarket(marketCode, candleType)
  │
  ├─ strategyConfigRepository.findByActiveTrue()
  │   → 활성 전략 설정 목록 (strategy_configs 테이블)
  │
  ├─ candleRepository.findLatestCandles(marketCode, candleType, 200)
  │   → 최근 200개 캔들 데이터
  │
  └─ 각 전략 설정에 대해:
       │
       ├─ strategyFactory.create(config)
       │   └─ strategyType별 분기:
       │       └─ "MA_CROSSOVER" → MovingAverageCrossoverStrategy(shortPeriod, longPeriod)
       │
       └─ strategy.analyze(marketCode, candles)
            │
            ├─ [MovingAverageCrossoverStrategy 기준]
            │   ├─ 캔들 시간순 정렬
            │   ├─ 현재 SMA(5), SMA(20) 계산
            │   ├─ 이전 SMA(5), SMA(20) 계산 (마지막 캔들 제외)
            │   │
            │   ├─ 골든크로스? (이전: 단기 <= 장기, 현재: 단기 > 장기)
            │   │   → StrategyResult.buy(confidence, "골든크로스")
            │   │
            │   ├─ 데드크로스? (이전: 단기 >= 장기, 현재: 단기 < 장기)
            │   │   → StrategyResult.sell(confidence, "데드크로스")
            │   │
            │   └─ 아무것도 아님?
            │       → StrategyResult.hold("교차 없음")
            │
            └─ [CompositeStrategy인 경우]
                ├─ AND 모드: 전체 전략이 동일 신호일 때만 실행
                ├─ OR 모드: 하나라도 신호 있으면 실행 (충돌 시 HOLD)
                └─ WEIGHTED 모드: 가중 점수 >= 0.3 이면 실행
```

### 4.3 매수 주문 흐름

```
TradingSignalEvent(BUY) 수신
  │
  └─ TradingCoordinator.onTradingSignal() (@EventListener)
       │
       ├─ 1. RiskManagementService.checkPreOrderRisk(marketCode, investAmount)
       │     │
       │     ├─ tradingHalted == true?
       │     │   → Rejected("전체 하드 리밋 도달로 매매가 중지되었습니다")
       │     │
       │     ├─ investAmount > 100,000원 (maxInvestmentPerTrade)?
       │     │   → Rejected("1회 투자 한도 초과")
       │     │
       │     ├─ calculateTotalBalance()
       │     │   └─ exchangeClient.getAccounts() → 전체 잔고 합산
       │     │   └─ 잔고 0원? → Rejected("잔고 조회 실패 또는 잔고 없음")
       │     │
       │     ├─ 포지션 비율 계산:
       │     │   openPositions.totalInvested + investAmount / totalBalance × 100
       │     │   > 50% (maxPositionRatio)?
       │     │   → Rejected("최대 포지션 비율 초과")
       │     │
       │     └─ 통과 → Approved
       │
       ├─ 2. Rejected → 로그 "[매수 거부] KRW-BTC 사유: ..." + 종료
       │
       └─ 3. Approved → OrderExecutionService.executeBuyOrder()
              │
              ├─ volume = investAmount / price (소수점 8자리)
              ├─ Order 생성 → orderRepository.save()
              │
              ├─ tradingConfig.mode 확인
              │   │
              │   ├─ PAPER 모드 (모의 매매):
              │   │   ├─ uuid = "PAPER-{timestamp}"
              │   │   ├─ state = DONE (즉시 체결)
              │   │   ├─ fee = price × volume × 0.0005
              │   │   └─ orderRepository.save()
              │   │
              │   └─ LIVE 모드 (실거래):
              │       ├─ tokenGenerator.generateToken(queryString)
              │       │   └─ JWT: access_key + nonce + query_hash(SHA512) + HS512 서명
              │       │
              │       ├─ exchangeClient.placeOrder()
              │       │   └─ orderRateLimiter(8/s) → CircuitBreaker
              │       │   └─ POST /orders {market, side=bid, ord_type=limit, price, volume}
              │       │       Authorization: Bearer {JWT}
              │       │
              │       ├─ order.updateFromExchange(uuid, state, executedVolume, paidFee)
              │       └─ orderRepository.save()
              │       └─ 실패 시: order.cancel() → save → 예외 throw
              │
              ├─ OrderExecutedEvent 발행
              │
              └─ order.isFilled() → Trade 기록
                  ├─ Trade 생성 → tradeRepository.save()
                  └─ TradeCompletedEvent(BID) 발행
                       │
                       ├─ [execution] PositionManagementService.onTradeCompleted()
                       │   └─ handleBuy(event)
                       │       ├─ 기존 OPEN 포지션 있음 → addVolume() (물타기) → save
                       │       └─ 없음 → Position(OPEN) 생성 → save
                       │                → PositionOpenedEvent 발행
                       │
                       └─ [monitoring] TradingEventListener.onTradeCompleted()
                            └─ alertManagementService.sendAlert()
                                 ├─ Discord: "매수 체결: KRW-BTC" (파란색)
                                 └─ alert_history 테이블 저장
```

### 4.4 매도 주문 흐름

```
TradingSignalEvent(SELL) 수신
  │
  └─ TradingCoordinator.onTradingSignal()
       │
       ├─ positionRepository.findByMarketCodeAndStatus(marketCode, OPEN)
       ├─ 포지션 없음 → 스킵
       └─ 포지션 있음 → OrderExecutionService.executeSellOrder()
              │
              └─ (매수와 동일한 흐름: PAPER/LIVE 분기)
                  │
                  └─ TradeCompletedEvent(ASK) 발행
                       │
                       ├─ [execution] PositionManagementService.onTradeCompleted()
                       │   └─ handleSell(event)
                       │       ├─ OPEN 포지션 찾기
                       │       ├─ position.close(sellPrice, CLOSED)
                       │       ├─ realizedProfit 계산
                       │       ├─ positionRepository.save()
                       │       └─ PositionClosedEvent 발행
                       │
                       └─ [monitoring] TradingEventListener
                            └─ Discord: "매도 체결: KRW-BTC" (파란색)
```

### 4.5 강제 청산 (리스크 발동)

```
TickerUpdatedEvent 수신 (WebSocket 실시간 시세)
  │
  └─ RiskManagementService.onTickerUpdated()
       │
       ├─ 포지션별 손절 체크
       │   수익률 = (현재가 - 평단가) / 평단가
       │   수익률 <= -3% → RiskBreachedEvent(STOP_LOSS)
       │
       ├─ 포지션별 익절 체크
       │   수익률 >= +5% → RiskBreachedEvent(TAKE_PROFIT)
       │
       └─ 전체 하드리밋 체크
           전체 미실현 손익 / 전체 투자금 × 100
           <= -15% → RiskBreachedEvent(HARD_LIMIT_TOTAL)
                      + tradingHalted = true (전면 중지)
       │
       └─ RiskBreachedEvent 수신처 (3곳 동시):
            │
            ├─ [execution] TradingCoordinator.onRiskBreached()
            │   ├─ STOP_LOSS, TAKE_PROFIT → executeForceSell()
            │   │   └─ 현재 수익률로 가격 추정 → executeSellOrder()
            │   │       strategyName = "RISK_STOP_LOSS" 또는 "RISK_TAKE_PROFIT"
            │   └─ HARD_LIMIT_TOTAL → 로그만 출력 (매매 이미 중지됨)
            │
            ├─ [execution] PositionManagementService.onRiskBreached()
            │   └─ STOP_LOSS, TAKE_PROFIT → closePosition() 대기
            │
            └─ [monitoring] RiskEventListener.onRiskBreached()
                 ├─ STOP_LOSS → Discord WARNING (주황색)
                 ├─ TAKE_PROFIT → Discord INFO (파란색)
                 ├─ HARD_LIMIT_MARKET → Discord ERROR (빨간색)
                 ├─ HARD_LIMIT_TOTAL → Discord CRITICAL (빨간색)
                 └─ MAX_POSITION_RATIO → Discord WARNING (주황색)
```

```

  │
  ├─ 1. AlertHistory 도메인 객체 생성
  │
  ├─ 2. NotificationSender 호출 (level별 분기)
  │     ├─ INFO     → send()        파란색 (0x3498DB)
  │     ├─ WARNING  → sendWarning() 주황색 (0xF39C12)
  │     ├─ ERROR    → sendError()   빨간색 (0xE74C3C)
  │     └─ CRITICAL → sendError()   빨간색 (0xE74C3C)
  │
  ├─ 3. DiscordWebhookSender.sendEmbed()
  │     ├─ 설정 미완료 시 → 자동 스킵 (isAlertsConfigured/isErrorsConfigured)
  │     ├─ alerts-webhook-url: 일반 알림 (INFO, WARNING)
  │     ├─ errors-webhook-url: 에러 알림 (ERROR, CRITICAL)
  │     └─ POST {webhookUrl}
  │         Body: {embeds: [{title, description, color, footer: "Bitcoin Auto Trader", timestamp}]}
  │     ├─ 성공 → 로그
  │     └─ 실패 → 예외 없이 로그만 출력
  │
  └─ 4. alert.markSent() → alertHistoryJpaRepository.save() → DB 기록
```

## 6. 백테스팅 흐름

```
사용자 → GET /backtest (Thymeleaf UI 페이지)
  │
  ├─ 파라미터 입력:
  │   ├─ 종목 (KRW-BTC)
  │   ├─ 전략 (MA_CROSSOVER)
  │   ├─ 캔들 타입 (DAYS / MINUTES_60)
  │   ├─ 기간 (시작일 ~ 종료일)
  │   ├─ 초기 잔고 (기본 100만원), 1회 투자금 (기본 10만원)
  │   ├─ 손절 -3% / 익절 +5%
  │   └─ MA 단기(5) / 장기(20) 기간
  │
  └─ POST /api/backtest/run (BacktestRq JSON)
       │
       └─ BacktestService.runAndSave()
            │
            ├─ 1. 캔들 데이터 로드
            │   candleRepository.findByMarketCodeAndCandleTypeAndDateRange()
            │   → 없으면 IllegalArgumentException
            │
            ├─ 2. StrategyConfig 생성 → StrategyFactory.create()
            │
            └─ 3. BacktestEngine.run(candles)
                 │
                 ├─ 캔들 시간순 정렬 → 하나씩 순회:
                 │
                 │   [포지션 보유 중이면]
                 │   ├─ profitRate = (현재가 - 평단가) / 평단가
                 │   ├─ <= stopLossThreshold(-3%) → 손절 매도 (loseTrades++)
                 │   └─ >= takeProfitThreshold(+5%) → 익절 매도 (winTrades++)
                 │
                 │   [전략 분석]
                 │   ├─ strategy.analyze(marketCode, window) → BUY/SELL/HOLD
                 │   ├─ BUY + 미보유 + 잔고 충분
                 │   │   → 투자금에서 수수료(0.05%) 차감 → volume 계산 → 매수
                 │   ├─ SELL + 보유 중
                 │   │   → 매도 (수수료 차감) → 승패 판정
                 │   └─ HOLD → 무시
                 │
                 │   [매 캔들마다]
                 │   └─ equity = 잔고 + (보유수량 × 현재가) → equityCurve 기록
                 │       peakEquity 갱신 → MDD 계산
                 │
                 ├─ 미청산 포지션 → 마지막 종가로 강제 정리
                 │
                 └─ BacktestResult 반환
                      ├─ totalReturn (수익률 %)
                      ├─ maxDrawdown (MDD %)
                      ├─ winRate (승률 %)
                      ├─ totalTrades, winTrades, loseTrades
                      ├─ trades (개별 거래 목록)
                      └─ equityCurve (자산 곡선)
       │
       ├─ backtest_results 테이블에 저장 (parameters, tradeDetails는 jsonb)
       └─ JSON 응답 → 프론트에서 Chart.js / Lightweight Charts로 시각화
```

### 백테스트 API

| 엔드포인트 | 설명 |
|-----------|------|
| `GET /backtest` | 백테스팅 UI 페이지 (Thymeleaf) |
| `POST /api/backtest/run` | 백테스팅 실행 + 결과 저장 |
| `GET /api/backtest/results` | 전체 결과 목록 (최신순) |
| `GET /api/backtest/results/{id}` | 특정 결과 상세 |
