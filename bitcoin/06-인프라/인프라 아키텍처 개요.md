### 이미지 흐름

```
kaniko (Step Pod) ──push──→ Registry (ClusterIP :5000 + NodePort :31500)
                                 │
K3s containerd ──pull──→ localhost:31500 (registries.yaml 미러 설정)
```

### 매니페스트 업데이트 흐름

```
Woodpecker Step 2 (alpine/git, SSH 인증)
  ├── liooos-k8s-manifests 레포 clone (SSH)
  ├── kustomization.yaml의 newTag를 전체 commit SHA로 수정
  ├── git commit + push
  └── ArgoCD가 변경 감지 → bitcoin 네임스페이스에 Sync (Rolling Update)
```

## 2. 왜 이렇게 구성하는가?

### 2.1 왜 K3s인가?

K3s는 Rancher Labs에서 만든 **경량 Kubernetes 배포판**이다.

| 비교 항목 | 표준 K8s (kubeadm) | K3s |
|-----------|-------------------|-----|
| 메모리 사용 | 1~2GB+ | ~512MB |
| 바이너리 크기 | 여러 개 분리 | 단일 ~100MB |
| etcd | 별도 설치 | 내장 SQLite (또는 etcd) |
| 인증서 관리 | 수동 | 자동 |
| 기본 인그레스 | 없음 | Traefik 내장 |
| 스토리지 | 별도 설치 | local-path-provisioner 내장 |
| 설치 시간 | 30분+ | 30초 |

**싱글 노드에 K3s를 선택한 이유:**
- 표준 K8s와 100% 호환되는 API (kubectl, Helm 모두 사용 가능)
- 메모리/CPU 오버헤드가 적어 앱에 자원을 더 할당 가능
- 배터리가 포함 (Traefik, CoreDNS, 스토리지 등 기본 내장)
- 프로덕션에서도 사용 가능한 CNCF 인증 배포판

### 2.2 왜 Woodpecker CI인가?

| 비교 | GitHub Actions | Jenkins | **Woodpecker CI** |
|------|---------------|---------|-------------------|
| 비용 | Private 월 2,000분 | 무료 | **무료** |
| 메모리 | (클라우드) | ~1GB+ | **~100MB** |
| Container Registry | ghcr.io 필요 | 필요 | **Local Registry (~30MB)** |
| 빌드 시각화 UI | GitHub 웹 | Jenkins 웹 | **Woodpecker 웹** |
| 파이프라인 정의 | YAML | Groovy | **YAML** |
| 설치 | (클라우드) | 높음 | **Helm chart (K3s Pod)** |
| GitHub 연동 | 네이티브 | 플러그인 | **OAuth 내장** |

**Woodpecker CI 선택 이유:**
1. **빌드 과정을 웹 UI에서 실시간 모니터링** (AWS CodeBuild처럼)
2. Go 네이티브 바이너리로 매우 가벼움 (~100MB RAM)
3. YAML 파이프라인 (GitHub Actions와 유사한 문법)
4. GitHub Webhook 수신을 내장 — 별도 webhook listener 불필요
5. 빌드 히스토리, 성공/실패 추적, 로그 스트리밍 내장
6. Codeberg(대형 Git 호스팅)의 메인 CI/CD로 사용 중 — 프로덕션 검증됨
7. **K8s 백엔드** 지원 — 파이프라인 Step이 각각 Pod으로 실행, 호스트에 Docker/JDK 불필요

### 2.3 왜 Synology 리버스 프록시인가?

GitHub Webhook이 미니 PC에 도달하려면 **공개 엔드포인트**가 필요하다.
이미 Synology NAS가 있으므로 추가 도구 없이 해결할 수 있다.

```
GitHub Webhook → yourname.synology.me → Synology 리버스 프록시 → 미니 PC
```

| 방법 | 비용 | 추가 설치 | 이미 있는가? |
|------|------|----------|------------|
| Cloudflare Tunnel | 도메인 필요 | cloudflared | 없음 |
| ngrok | 무료 (제한) | ngrok | 없음 |
| **Synology 리버스 프록시** | **무료** | **없음** | **있음** |

**Synology가 이미 제공하는 것:**
- DDNS (동적 IP 대응) — `yourname.synology.me`
- HTTPS (Let's Encrypt 자동 갱신)
- 리버스 프록시 (내부 IP + 포트로 전달)

> 추후 도메인을 구매하면 Cloudflare Tunnel로 전환하거나
> Synology에서 커스텀 도메인으로 DDNS를 변경할 수 있다.

### 2.4 왜 ArgoCD를 사용하는가?

이전에는 Woodpecker CI가 `kubectl set image`로 직접 배포했다. 이 방식은 단순하지만 **Git에 배포 상태가 기록되지 않는** 단점이 있었다. ArgoCD를 도입하면서 **GitOps** 방식으로 전환했다.

| 비교 항목 | kubectl set image (이전) | **ArgoCD + 매니페스트 레포 (현재)** |
|-----------|------------------------|----------------------------------|
| 배포 상태 추적 | kubectl로만 확인 | **Git 히스토리에 기록** |
| 롤백 | kubectl rollout undo | **Git revert → 자동 반영** |
| 선언적 관리 | X | **O (Git = 진실의 원천)** |
| 배포 시각화 | Woodpecker UI (빌드만) | **ArgoCD UI (배포 상태 + 리소스 현황)** |
| RBAC 복잡도 | deploy ServiceAccount 필요 | **ArgoCD가 관리 (RBAC 단순화)** |

**이전에 ArgoCD를 제외했던 이유와 해결:**

| 이전 우려 사항 | 해결 방법 |
|--------------|----------|
| Git 히스토리 오염 (이미지 태그 커밋) | **별도 매니페스트 레포** 분리 → 소스 레포 히스토리 깔끔 유지 |
| 오버엔지니어링 (~512MB RAM) | 실제 ~350-500MB 수준, 32GB 대비 미미. GitOps 이점이 더 큼 |

> **핵심 변경**: CI(빌드)는 Woodpecker, CD(배포)는 ArgoCD로 역할을 분리했다.
> Woodpecker는 빌드 후 매니페스트 레포의 이미지 태그만 업데이트하고,
> ArgoCD가 변경을 감지하여 자동으로 배포한다.

### 2.5 왜 별도 매니페스트 레포인가?

소스 코드와 K8s 매니페스트를 같은 레포에 두면 이미지 태그 업데이트 커밋이 소스 코드 히스토리에 섞인다.

| 방식 | 장점 | 단점 |
|------|------|------|
| 소스+매니페스트 동일 레포 | 단순 | **Git 히스토리 오염**, CI 무한 루프 위험 |
| **별도 매니페스트 레포** | **히스토리 분리**, CI 트리거 독립 | 레포 2개 관리 |

**매니페스트 레포 (`liooos-k8s-manifests`) 구조:**
- `postgres/base/` — 공용 PostgreSQL 매니페스트 (`database` 네임스페이스)
- `postgres/overlays/prod/` — PostgreSQL 프로덕션 오버라이드
- `bitcoin/base/` — 환경 공통 K8s 매니페스트 (Deployment, Service, ConfigMap 등)
- `bitcoin/overlays/prod/` — 프로덕션 오버라이드 (이미지 태그 등)
- `bitcoin/base/app-sealed-secret.yaml` — 암호화된 SealedSecret (Kustomize 경로 제약으로 base/ 내 위치)
- `postgres/base/postgres-sealed-secret.yaml` — PostgreSQL 암호화된 SealedSecret
- `argocd/` — ArgoCD Application 정의 (bitcoin-app, postgres-app)

### 2.6 왜 Local Registry인가?

| 방식 | 장점 | 단점 |
|------|------|------|
| `docker save \| k3s ctr import` | Registry 불필요 | 호스트에 Docker 필요, 멀티 노드 미대응 |
| ghcr.io | 표준 | Private 용량 제한, 네트워크 의존 |
| **Local Registry** | **K3s 내에서 완결**, 멀티 노드 즉시 대응 | ~30MB 추가 |

**Local Registry 선택 이유:**
1. 호스트에 Docker 설치 불필요 — K3s만으로 모든 것 동작
2. kaniko가 Registry에 push → K3s containerd가 pull → 표준 워크플로
3. 멀티 노드 확장 시 아무 변경 없이 그대로 사용 가능
4. 리소스 ~30MB로 미미

### 2.7 왜 Sealed Secrets인가?

Kubernetes Secret은 **base64 인코딩일 뿐 암호화가 아니다**.

```
# 일반 Secret → base64 디코딩하면 평문 노출!
apiVersion: v1
kind: Secret
data:
  api-key: dXBiaXQtc2VjcmV0LWtleQ==   ← echo '...' | base64 -d → upbit-secret-key (노출!)
```

Secret을 Git에 커밋하면 API 키가 노출된다. Sealed Secrets는:
1. 로컬에서 `kubeseal`로 **암호화**된 SealedSecret 생성
2. 암호화된 SealedSecret만 Git에 커밋 (안전)
3. 클러스터의 Sealed Secrets Controller가 복호화 → 일반 Secret 생성

## 3. 데이터 흐름 (배포 사이클)

```
① 개발자가 코드 수정 후 GitHub dev 브랜치에 push

② GitHub Webhook → Synology 리버스 프록시 → Woodpecker CI Server (K3s Pod)
   └── Woodpecker가 GitHub OAuth로 인증, push 이벤트 수신

③ Woodpecker Agent가 K8s 백엔드로 Step Pod 생성 (웹 UI에서 실시간 모니터링)
   ├── Step 1 Pod: kaniko → 멀티스테이지 Dockerfile로 Gradle 빌드 + 이미지 패키징 → Local Registry push
   └── Step 2 Pod: alpine/git → SSH로 매니페스트 레포(liooos-k8s-manifests) clone
                   → kustomization.yaml의 newTag를 전체 commit SHA로 업데이트 → git push

④ ArgoCD가 매니페스트 레포 변경 감지 → 자동 Sync
   └── Kustomize로 렌더링 → bitcoin 네임스페이스에 kubectl apply

⑤ K8s Rolling Update (무중단 배포)
   ├── 새 Pod 생성 (Registry에서 이미지 pull)
   ├── readinessProbe 통과 대기
   ├── 트래픽을 새 Pod으로 전환
   └── 이전 Pod graceful shutdown 후 종료

⑥ 배포 완료
   ├── Woodpecker UI에서 빌드 성공/실패 확인
   ├── ArgoCD UI에서 배포 상태/리소스 현황 확인
   └── Discord 알림 (성공/실패)
```

## 4. 무중단 배포 전략

### 4.1 Rolling Update

```yaml
# K8s Deployment 설정
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # 새 Pod 최대 1개 추가 생성
      maxUnavailable: 0    # 기존 Pod 0개 다운 → 무중단 보장
```

**흐름:**
```
시작 상태:  [App v1 Pod] ← 트래픽 수신 중

Step 1:     [App v1 Pod] [App v2 Pod 생성 중...]
            ↑ 트래픽      readinessProbe 대기

Step 2:     [App v1 Pod] [App v2 Pod ✅ Ready]
            ↑ 트래픽       ↑ 트래픽 (둘 다 수신)

Step 3:     [App v1 Pod 종료 중...] [App v2 Pod]
            graceful shutdown       ↑ 트래픽

완료 상태:  [App v2 Pod] ← 트래픽 수신 중
```

> 무중단 배포는 **K8s 자체 기능**이다. CI/CD 도구(Woodpecker, ArgoCD)가 뭐든 상관없이 동일하게 동작한다.

### 4.2 Health Check (readinessProbe / livenessProbe)

```yaml
containers:
  - name: app
    readinessProbe:            # "트래픽을 받을 준비가 되었는가?"
      httpGet:
        path: /actuator/health
        port: 9000
      initialDelaySeconds: 30  # 앱 시작 후 30초 뒤부터 체크
      periodSeconds: 10        # 10초마다 확인
    livenessProbe:             # "앱이 살아있는가?"
      httpGet:
        path: /actuator/health
        port: 9000
      initialDelaySeconds: 60  # 시작 후 60초 뒤부터 체크
      periodSeconds: 15        # 15초마다 확인
```

### 4.3 Spring Boot Graceful Shutdown

```yaml
# application.yml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

K8s가 Pod 종료 시:
1. SIGTERM 전송
2. Spring Boot가 진행 중인 요청 완료 대기 (최대 30초)
3. 새 요청은 받지 않음
4. 모든 요청 완료 후 종료

## 9. 고가용성(HA) 및 확장 로드맵

### 9.1 무중단 배포의 한계

현재 `replicas: 1`에서 Rolling Update(`maxSurge:1, maxUnavailable:0`)는 **배포 순간에만** 일시적으로 Pod 2개를 운영한다.

```
배포 시작:  [v1 Pod] ← 트래픽             (1개)
Step 1:    [v1 Pod] [v2 Pod 생성 중]       (2개 — maxSurge:1)
Step 2:    [v1 Pod] [v2 Pod ✅ Ready]      (2개, 둘 다 트래픽)
Step 3:    [v1 종료] [v2 Pod]              (1개로 복귀)
```

| 상황 | replicas:1 | replicas:2 | 멀티 노드 |
|------|:----------:|:----------:|:--------:|
| 배포 중 무중단 | O | O | O |
| Pod 크래시 대응 | **X** (다운타임) | O | O |
| 노드 장애 대응 | **X** | **X** | O |

### 9.2 단계별 확장 로드맵

| 단계 | 구성 | 대응 범위 | 추가 비용 | 아키텍처 변경 |
|------|------|----------|----------|-------------|
| **1단계 (현재)** | 1노드, replicas:1 | 배포 시 무중단 | 0원 | - |
| **2단계** | 1노드, replicas:2 | + Pod 크래시 대응 | 0원 (RAM 여유) | Deployment replicas 변경만 |
| **3단계** | 2노드, replicas:2 | + 노드 장애 대응 | 미니 PC 1대 | 스토리지 변경 필요 |
| **4단계** | 3노드, replicas:3 | + 완전 HA | 미니 PC 2대 | 동일 |

> Local Registry 기반이므로 **멀티 노드 전환 시 이미지 배포 방식은 변경 불필요**. Registry가 이미 NodePort로 노출되어 있으므로 워커 노드에서도 접근 가능하다.

### 9.3 2단계: replicas 증가 (싱글 노드)

변경 사항이 거의 없다. 매니페스트 레포에서 Deployment의 replicas만 수정하면 ArgoCD가 자동 반영한다.

```yaml
# liooos-k8s-manifests/bitcoin/base/app-deployment.yaml
spec:
  replicas: 2    # 1 → 2로 변경
```

리소스 영향:

| replicas | App 메모리 사용 | 전체 사용량 |
|----------|:-------------:|:---------:|
| 1 | 512MB~1GB | ~5.0GB |
| 2 | 1GB~2GB | ~6.0GB |
| 3 | 1.5GB~3GB | ~7.0GB |

> 32GB 기준 replicas:3이어도 여유 25GB+ → **문제없음**

### 9.4 3단계: 멀티 노드 (워커 노드 추가)

K3s는 워커 노드를 **한 줄 명령어로** 추가할 수 있다. 한 대씩 자유롭게 늘릴 수 있다.

```bash
# 미니 PC 2에서 실행 (워커 노드로 참가)
curl -sfL https://get.k3s.io | K3S_URL=https://미니PC1:6443 K3S_TOKEN=<토큰> sh -
```

#### 멀티 노드 구성도

```
미니 PC 1 (마스터)                    미니 PC 2 (워커)
┌──────────────────────────┐        ┌──────────────────────────┐
│ K3s Server               │        │ K3s Agent                │
│                          │        │                          │
│ [App Pod 1]              │        │ [App Pod 2]              │
│ [PostgreSQL]             │        │                          │
│ [Woodpecker CI]          │        │                          │
│ [ArgoCD]                 │        │                          │
│ [Local Registry :31500]  │ ←pull─ │                          │
│ [Sealed Secrets]         │        │                          │
└──────────────────────────┘        └──────────────────────────┘
         ↑                                    ↑
         └────── Service (트래픽 분산) ──────────┘
```

#### 아키텍처 변경이 필요한 부분

| 항목 | 싱글 노드 (현재) | 멀티 노드 (변경) | 변경 난이도 |
|------|:---------------:|:-------------:|:---------:|
| **이미지 배포** | Local Registry | **변경 없음** (이미 Registry 사용 중) | 없음 |
| **Woodpecker 파이프라인** | 동일 | **변경 없음** | 없음 |
| **ArgoCD 배포** | 동일 | **변경 없음** | 없음 |
| **PostgreSQL 스토리지** | local-path (노드 종속) | Longhorn 또는 NFS | 높음 |
| **Pod 스케줄링** | 설정 불필요 | podAntiAffinity 추가 | 낮음 |

#### Pod Anti-Affinity

같은 앱의 Pod이 **서로 다른 노드에 분산**되도록 설정한다.

```yaml
# liooos-k8s-manifests/bitcoin/base/app-deployment.yaml에 추가
spec:
  template:
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                labelSelector:
                  matchLabels:
                    app: bitcoin-trader
                topologyKey: kubernetes.io/hostname
```

> `preferred` = 가능하면 분산 (노드 부족 시 같은 노드에 배치 허용)
> `required` = 반드시 분산 (노드 부족 시 Pod 생성 불가)

#### PostgreSQL 스토리지 (멀티 노드 시 주의)

현재 `local-path-provisioner`는 **특정 노드의 로컬 디스크에 데이터를 저장**한다.
멀티 노드에서 PostgreSQL Pod이 다른 노드로 이동하면 **데이터에 접근 불가**.

해결 방법:
- **Longhorn** (K3s 권장 분산 스토리지) — 노드 간 데이터 복제
- **NFS** — Synology NAS를 NFS 서버로 활용 (이미 있으므로 추가 비용 0)
- **nodeSelector** — PostgreSQL을 항상 마스터 노드에 고정 (가장 단순)

### 9.5 현재 권장

1단계(현재)에서 **2단계(replicas:2)로 전환**을 권장한다.
- 변경: 매니페스트 레포에서 Deployment replicas를 1 → 2로 수정 (한 줄) → ArgoCD 자동 반영
- 효과: Pod 크래시 시에도 서비스 무중단
- 비용: RAM ~512MB 추가 (32GB 중 미미)

3단계(멀티 노드)는 필요할 때 전환하면 된다. Local Registry 기반이므로 **이미지 배포 방식 변경 없이** 워커 노드를 추가하면 된다.

## 11. 핵심 개념 용어 정리

| 용어 | 설명 |
|------|------|
| **Pod** | K8s 최소 배포 단위. 1개 이상의 컨테이너를 포함 |
| **Deployment** | Pod의 생성/업데이트/스케일링을 관리. Stateless 앱에 적합 |
| **StatefulSet** | 안정적인 네트워크 ID와 영속적 스토리지가 필요한 앱용 (DB 등) |
| **Service** | Pod에 안정적인 네트워크 엔드포인트 제공 (Pod IP는 변경될 수 있으므로) |
| **Ingress** | 외부 HTTP(S) 트래픽을 클러스터 내 Service로 라우팅하는 규칙 |
| **ConfigMap** | 환경변수, 설정 파일 등 비밀이 아닌 설정 데이터 저장 |
| **Secret** | 패스워드, API 키 등 민감 데이터 저장 (base64 인코딩) |
| **PVC** | PersistentVolumeClaim. 영속적 스토리지를 요청하는 추상화 |
| **Namespace** | 클러스터 내 리소스를 논리적으로 분리하는 가상 그룹 |
| **Kustomize** | K8s 매니페스트를 환경별로 커스터마이징하는 도구 (kubectl 내장) |
| **Rolling Update** | 새 버전 Pod을 점진적으로 배포하는 무중단 배포 전략 |
| **readinessProbe** | Pod이 트래픽을 받을 준비가 되었는지 확인하는 헬스체크 |
| **Graceful Shutdown** | 진행 중인 요청을 완료한 후 안전하게 종료하는 방식 |
| **Woodpecker CI** | Go 기반 경량 CI 엔진. K8s 백엔드, YAML 파이프라인 (빌드 담당) |
| **ArgoCD** | GitOps 기반 CD 도구. Git 레포를 감시하고 K8s에 자동 배포 |
| **GitOps** | Git을 단일 진실의 원천으로 사용하는 운영 방식. Git 변경 = 배포 |
| **매니페스트 레포** | K8s 매니페스트만 관리하는 별도 Git 저장소. 소스 레포와 분리하여 히스토리 오염 방지 |
| **kaniko** | Docker 데몬 없이 컨테이너 내에서 이미지를 빌드하는 도구 |
| **Local Registry** | 클러스터 내에서 운영하는 컨테이너 이미지 저장소 (registry:2) |
| **replicas** | Deployment에서 유지할 Pod 개수. 이중화 및 부하 분산에 사용 |
| **podAntiAffinity** | 같은 앱의 Pod을 서로 다른 노드에 분산 배치하는 정책 |
| **Longhorn** | K3s 권장 분산 스토리지. 노드 간 데이터 복제 지원 |
