## 1. 시스템 개요

Upbit API 기반 암호화폐 자동매매 시스템.
멀티모듈 모노리스 구조의 Spring Boot 애플리케이션으로, 이벤트 기반 아키텍처와 헥사고날 아키텍처를 적용한다.

### 핵심 목표
- 전략 플러그인 방식으로 매매 전략 조합/분리
- Paper Trading을 통한 전략 실시간 검증
- 백테스팅 + 시각화
- 리스크 관리 (손절/익절/하드리밋)
- 24시간 무중단 자동 운영
- MSA 전환 대비 모듈 격리 + Outbox 패턴

## 5. 모듈 의존성 규칙

```
core ← api-client ← strategy
  ↑
  ├── execution (core, api-client 의존)
  └── monitoring (core만 의존)

app → 전체 모듈 조립 + Outbox 인프라
```

| 모듈 | 허용된 의존 | 격리 방법 |
|------|------------|----------|
| `core` | 없음 (순수 도메인) | Gradle `dependencies` |
| `api-client` | core | Gradle + ArchUnit |
| `strategy` | core, api-client | Gradle + ArchUnit |
| `execution` | core, api-client | Gradle + ArchUnit |
| `monitoring` | core | Gradle + ArchUnit |
| `app` | 전체 (조립 전용) | - |

### ArchUnit 검증 (6개 테스트)

| 테스트 | 규칙 |
|--------|------|
| core 독립성 | core는 다른 모듈에 의존하지 않아야 한다 |
| monitoring 격리 | monitoring은 core만 의존해야 한다 |
| strategy 격리 | strategy는 core와 apiclient만 의존해야 한다 |
| execution 격리 | execution은 core와 apiclient만 의존해야 한다 |
| 헥사고날 규칙 | domain은 infrastructure/adapter에 의존하지 않아야 한다 |
| 예외 상속 | 커스텀 예외는 DomainException을 상속해야 한다 |

## 6. 이벤트 아키텍처

### 6.1 이벤트 분류

#### Outbox 대상 (비즈니스 이벤트 — DomainEventPublisher 사용)

| 이벤트 | 발행 모듈 | 소비 모듈 | 용도 |
|--------|----------|----------|------|
| `OrderExecutedEvent` | execution | monitoring | 주문 체결 |
| `TradeCompletedEvent` | execution | execution, monitoring | 거래 완료 → 포지션 관리, 알림 |
| `PositionOpenedEvent` | execution | monitoring | 포지션 개설 |
| `PositionClosedEvent` | execution | monitoring | 포지션 종료 |
| `RiskBreachedEvent` | execution | execution, monitoring | 리스크 위반 → 강제 매도, 알림 |
| `TradingSignalEvent` | strategy | execution | 매매 신호 |
| `DepositDetectedEvent` | - | monitoring | 입금 감지 |
| `WithdrawalDetectedEvent` | - | monitoring | 출금 감지 |

#### 비-Outbox (실시간/시스템 이벤트 — ApplicationEventPublisher 직접 사용)

| 이벤트 | 발행 모듈 | 용도 | 이유 |
|--------|----------|------|------|
| `TickerUpdatedEvent` | api-client | 실시간 시세 | 고빈도, DB 부하 과다 |
| `MarketDataReceivedEvent` | api-client | 캔들 수집 알림 | 유실 허용 |
| `SystemErrorEvent` | api-client | 시스템 에러 | 일시적, 유실 허용 |
| `AlertEvent` | - | 일반 알림 | 유실 허용 |

### 6.2 Outbox 패턴 동작

```
Service (비즈니스 로직)
  ↓ DomainEventPublisher.publish(event)
  ↓
OutboxDomainEventPublisher
  ├── 1. OutboxEvent 저장 (같은 트랜잭션, 원자성 보장)
  └── 2. ApplicationEventPublisher.publishEvent() (모놀리스 내 즉시 전달)

OutboxEventProcessor (스케줄러)
  ├── 1분 간격: PENDING → PROCESSED 마킹 (현재: 모놀리스이므로 이미 전달 완료)
  └── 5분 간격: FAILED 이벤트 재시도 (최대 3회)
```

**MSA 전환 시**: OutboxEventProcessor에서 HTTP/MQ로 외부 전송 후 PROCESSED 마킹으로 변경.

### 6.3 실시간 매매 흐름

```
WebSocket → TickerUpdatedEvent
  → StrategyEvaluationService (전략 분석) [미구현]
    → TradingSignalEvent (BUY/SELL/HOLD)
      → TradingCoordinator (@EventListener)
        → RiskManagementService.checkPreOrderRisk()
          ├─ Approved → OrderExecutionService (LIVE/PAPER 분기)
          │              → OrderExecutedEvent → DomainEventPublisher → Outbox + 즉시 전달
          │              → TradeCompletedEvent → PositionManagementService + Discord 알림
          └─ Rejected → 로그 (거부 사유)
```

### 6.4 리스크 모니터링 흐름

```
TickerUpdatedEvent → RiskManagementService (@EventListener)
  → 손절(-3%)/익절(+5%)/하드리밋(-15%) 체크
    → RiskBreachedEvent → DomainEventPublisher → Outbox
      → TradingCoordinator (강제 매도) + Discord 알림
```

## 8. 디자인 패턴

| 패턴 | 적용 위치 | 목적 |
|------|----------|------|
| Strategy | `TradingStrategy` 인터페이스 | 매매 전략 플러그인 |
| Composite | `CompositeStrategy` | AND/OR/가중치 전략 조합 |
| Factory | `StrategyFactory` | JSONB 파라미터 → 전략 인스턴스 |
| Outbox | `OutboxDomainEventPublisher` | 이벤트 영속화 + 원자성 보장 |
| Ports & Adapters | 전 모듈 | 도메인 ↔ 인프라 분리 |
| Rate Limiter | Resilience4j | API 초당 호출 제한 준수 |
| Circuit Breaker | Resilience4j | API 장애 시 안전 모드 |
| Event-Driven | Spring ApplicationEvent | 모듈 간 느슨한 결합 |
| Aggregate Root | Market, Order, Position | DDD 집합 루트 |
| Value Object | Money, Percentage, MarketCode | 불변 값 객체 |
| Sealed Class | `RiskCheckResult` | 타입 안전 결과 표현 |
