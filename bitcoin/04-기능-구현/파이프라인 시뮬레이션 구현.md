# 파이프라인 시뮬레이션 구현

## 개요

수집된 **1분봉 캔들 데이터**를 실제 프로덕션 파이프라인 코드(전략 분석 → 리스크 체크 → 주문 실행 → 포지션 관리)에 빠르게 통과시켜 시뮬레이션하는 기능.

기존 `BacktestEngine`과 달리 **실제 서비스 클래스**를 그대로 사용하여 프로덕션 코드 경로를 검증한다.

## 기존 BacktestEngine vs PipelineSimulator

| | BacktestEngine | PipelineSimulator |
|--|---|---|
| **위치** | strategy 모듈 | app 모듈 |
| **매매 로직** | 자체 단순화 로직 | 프로덕션 서비스 코드 경유 |
| **사용 서비스** | 없음 (자체 구현) | OrderExecutionService, TradingCoordinator, RiskManagementService, PositionManagementService |
| **리스크 관리** | 손절/익절만 | 전체 (손절, 익절, 포지션 비율, 하드리밋) |
| **포지션 관리** | 단순 보유량 추적 | Position 도메인 모델 + 상태 전이 |
| **이벤트 흐름** | 없음 | DomainEvent 기반 파이프라인 |
| **결과 데이터** | 거래 기록 + 자산곡선 | 거래 + 포지션 + 리스크이벤트 + 신호 + 자산곡선 |
| **용도** | 전략 성능 평가 | 파이프라인 전체 동작 검증 |

## 아키텍처

### 모듈 배치 결정

`PipelineSimulator`는 `execution` + `strategy` 모듈 모두에 의존하므로, **순환 의존성 방지**를 위해 `app` 모듈에 배치했다.

```
app/src/main/kotlin/com/autocoin/simulation/
├── SimulationRequest.kt          # 요청 모델
├── SimulationResult.kt           # 결과 모델 (거래/포지션/리스크/신호/자산곡선)
├── InMemoryRepositories.kt       # 인메모리 Order/Trade/Position 저장소
├── PipelineSimulator.kt          # 핵심 엔진 - 1분봉 순차 리플레이
├── SimulationService.kt          # 오케스트레이션 (CandleRepository → Simulator)
└── SimulationApiController.kt    # REST API
```

### 핵심 설계

```
CandleRepository에서 1분봉 조회
  └→ PipelineSimulator.run(candles)
       ├── 인메모리 저장소 생성 (Order, Trade, Position)
       ├── CollectingDomainEventPublisher (이벤트 수집)
       ├── SimulationExchangeClient (잔고 stub)
       ├── 실제 서비스 조립:
       │   ├── OrderExecutionService (Paper 모드)
       │   ├── RiskManagementService
       │   ├── TradingCoordinator
       │   └── PositionManagementService
       │
       └── 캔들 순차 리플레이:
           for (candle in sorted) {
             1. TickerUpdatedEvent → RiskManagement (손절/익절 체크)
             2. RiskBreachedEvent 처리 → 강제 매도
             3. Strategy.analyze() → 신호 판단
             4. TradingSignalEvent → TradingCoordinator → 주문 실행
             5. TradeCompletedEvent → PositionManagement
             6. 자산곡선 기록 + MDD 계산
           }
```

### 인메모리 저장소

Spring/JPA 없이 동작하기 위해 Port 인터페이스의 인메모리 구현체를 사용:

- `InMemoryOrderRepository` — 리플렉션으로 id 자동 생성
- `InMemoryTradeRepository`
- `InMemoryPositionRepository`

### 이벤트 수집

`CollectingDomainEventPublisher`가 이벤트를 리스트에 수집하고, 시뮬레이터가 직접 다음 리스너를 호출하는 방식으로 이벤트 파이프라인을 시뮬레이션한다.

```kotlin
class CollectingDomainEventPublisher(
    private val events: MutableList<AbstractDomainEvent>
) : DomainEventPublisher {
    override fun publish(event: AbstractDomainEvent) { events.add(event) }
}
```

## 결과 데이터 상세

| 항목 | 설명 |
|------|------|
| `trades` | 매매 기록 (번호, 매수/매도, 시각, 가격, 수량, 수수료, 잔고) |
| `positions` | 포지션 기록 (개설/종료 시각, 평단가, 매도가, 실현손익, 수익률, 종료 사유) |
| `riskEvents` | 리스크 발동 기록 (손절/익절/하드리밋, 현재률, 임계값) |
| `signals` | 전략 신호 기록 (BUY/SELL, 신뢰도, 실행 여부, 사유) |
| `equityCurve` | 자산곡선 (시각, 총자산, 현금, 포지션 가치) |

## 테스트 (PipelineSimulatorTest)

| 테스트 | 검증 내용 |
|--------|-----------|
| 골든크로스 매수/데드크로스 매도 | 패턴 감지 → 신호 발행 → 거래 실행 |
| 손절 발동 | 급락 시 리스크 이벤트 → 강제 매도 |
| 캔들 부족 시 무거래 | 5개 미만 캔들 → 신호 없음 |
| 자산곡선 기록 | 초기 자산 = initialBalance |
| 결과 요약 정확성 | 전략명, 마켓코드, 캔들 수 등 메타데이터 |
| 리스크 이벤트 기록 | 큰 하락 시 리스크 이벤트 수집 |
