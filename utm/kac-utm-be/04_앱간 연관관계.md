# 앱 간 연관관계

#kac-utm #backend #architecture

> 이 문서는 11개 app 모듈이 서로 어떻게 통신하는지를 정리한다. 신입 개발자가 "내가 수정한 코드가 어디에 영향을 미치는지" 판단할 때 참고.

---

## 1. 통신 방식 요약

| 통신 방식 | 프로토콜 | 특징 | 사용처 |
|-----------|---------|------|--------|
| **Feign Client** | HTTP (동기) | 선언적 REST 호출, 자동 재시도 | App 간 기본 통신 |
| **WebClient** | HTTP (비동기) | 리액티브, SSE 스트리밍 지원 | ws→engine, uss-pub→SWIM |
| **RabbitMQ** | AMQP | 메시지 큐, 느슨한 결합 | 드론 항적 데이터 수신 |
| **WebSocket** | WS | 실시간 양방향 | 클라이언트 실시간 데이터 |
| **MQTT** | MQTT | IoT 경량 프로토콜 | 드론 장비 통신 |
| **TCP** | TCP (Netty) | 바이너리/텍스트 소켓 | 외부 장비 직접 통신 |

## 4. RabbitMQ 메시지 흐름

### 큐 목록

| 큐 이름 | 소비자 | 데이터 | 설명 |
|---------|--------|--------|------|
| `q_smatii` | [[app-psty]] | SmatiiRq (LTEM 드론 항적) | SMATII 드론 위치 데이터 |
| `q_illegal` | [[app-psty]] | IllegalInboundSocketRq | 불법 드론 탐지 데이터 |
| `q_smatii_rnd` | [[app-uss-pub]] | SmatiiRq | USS용 드론 항적 데이터 |

### 흐름

```
외부 드론/스캐너
      │
      ▼
  RabbitMQ
  ┌───────────────────────────────┐
  │ q_smatii     → PSTY (관제용)  │
  │ q_illegal    → PSTY (불법드론) │
  │ q_smatii_rnd → USS-PUB (USS용)│
  └───────────────────────────────┘
```

### RabbitConsumer 처리 상세 (PSTY)

```java
// q_smatii 리스너
@RabbitListener(queues = "q_smatii")
queueSmatii(String message) {
    SmatiiRq → SocketPayload 변환
    → command.smatiiCommand(socketPayload)
}

// q_illegal 리스너
@RabbitListener(queues = "q_illegal")
illegalSmatii(String message) {
    IllegalInboundSocketRq → SocketPayload 변환
    → command.illegalCommand(socketPayload)
}
```

## 5. 드론 데이터 수신 → 관제 표시 전체 흐름

```
┌──────────────────────────────────────────────────────────────┐
│               드론 데이터 수신 → 관제 표시 흐름                  │
└──────────────────────────────────────────────────────────────┘

① 드론/스캐너에서 데이터 송신
   ├── SMATII (LTEM) → RabbitMQ (q_smatii)
   ├── 불법드론 스캐너 → RabbitMQ (q_illegal)
   └── MQTT 장비 → Bridge (MQTT Client)
                          │
② PSTY가 RabbitMQ에서 수신
   │  RabbitConsumer.queueSmatii() / illegalSmatii()
   │
③ SocketCommand 처리
   │  ├── 좌표 유효성 검사 (한국 범위 내?)
   │  ├── Control ID 캐시 조회
   │  │   └── 캐시 미스 → Engine API 호출
   │  │       POST /api/v1/engine/psty/ctrl/crt/id
   │  │       → 비행계획 매칭, 허가 여부 확인
   │  │       → Control ID 발급 + 캐시 저장
   │  │
④  │  GPCollection에 데이터 추가
   │  │  (메모리 버퍼, 20초마다 Engine으로 배치 업로드)
   │  │
⑤  │  WebSocket으로 실시간 전송
   │  └── POST /api/ws/receiver (WS Client)
   │
⑥ WS가 클라이언트(브라우저)에 WebSocket Push
   │
⑦ 관제 화면에 드론 위치 실시간 표시
```

## 6. 비행계획 신청 → 승인 전체 흐름

```
┌──────────────────────────────────────────────────────────────┐
│             비행계획 신청 → 승인 흐름                           │
└──────────────────────────────────────────────────────────────┘

① 사용자가 web/uss-pub에서 비행계획 신청
   │  POST /api/v1/web/fpl/bsc (web)
   │  POST /api/v1/web/fpl/bsc (uss-pub)
   │
② web/uss-pub → Engine Feign 호출
   │  EngineFplClient.crtFpl()
   │  POST /api/v1/engine/fpl
   │
③ Engine: 비행계획 검증
   │  FplBscService.crtFpl()
   │  ├── 비행영역 좌표 → Geometry 변환
   │  ├── Buffer(완충거리) 적용
   │  ├── ★ 공역 충돌 검사 ★
   │  │   getViolatedCmptncInstSpaces()
   │  │   비행영역 vs 관제권 Geometry 교차 판정
   │  │   → 충돌 영역별 검토대기(WAIT) 항목 생성
   │  ├── 조종사/기체 정보 저장
   │  └── 동적 스케줄러 등록
   │
④ 비행계획 상태: APPLY(신청) → REVIEW(검토중)
   │
⑤ 관리자가 mngr에서 비행계획 검토
   │  GET /api/v1/mngr/.../fpl/plist
   │  → EngineFplClient로 Engine 데이터 조회
   │
⑥ 관리자 승인/반려
   │  EngineFplClient.aprvPrmsn()
   │  PUT /api/v1/engine/fpl/prmsn/aprv
   │
⑦ 상태 변경: REVIEWED(검토완료)
   │  ├── 승인(APPROVED) → 비행 가능
   │  └── 반려(REJECTED) → 사유와 함께 통보
   │
⑧ 알림 발송 (SysAlrtService)
```

## 8. 핵심 정리: "이 앱을 수정하면 어디에 영향이 있나?"

| 수정 대상 | 직접 영향 | 간접 영향 |
|-----------|----------|----------|
| **engine** | web, mngr, sol, uss-pub (모두 Feign으로 호출) | ws, psty (engine이 호출) |
| **web** | 프론트엔드 | engine (Feign 호출) |
| **mngr** | 관리자 프론트엔드 | engine, batch, sol (Feign 호출) |
| **psty** | 드론 데이터 처리 중단 | ws (데이터 미전달), engine (이력 미저장) |
| **ws** | 실시간 화면 중단 | 클라이언트 WebSocket |
| **batch** | 통계 미생성, 계정 처리 중단 | mngr (배치 관리 UI) |
| **bridge** | 외부 장비 통신 중단 | psty (데이터 미수신) |
| **uss-pub** | USS 기능 중단 | engine, SWIM |
