# 통신 방식 비교: SSE vs WebSocket

> 작성일: 2026-03-19
> 상태: 방식 선택 전
> 관련 문서: [[00. 개요]]

---

## 후보 방식

| 방식 | 구조 | 이중화 방법 |
|------|------|-----------|
| **SSE + REST** | 전송=REST, 수신=SSE 스트림 | Redis Pub/Sub |
| **WebSocket + Redis** | 양방향 WS + Redis 중계 | Redis Pub/Sub + Sticky Session |
| **WebSocket + STOMP + RabbitMQ** | STOMP 프로토콜 + 외부 브로커 | RabbitMQ 릴레이 + Sticky Session |

## 기능 비교

| 항목 | SSE + REST | WebSocket |
|------|-----------|-----------|
| **통신 방향** | 단방향 (서버→클라이언트) + REST (클라이언트→서버) | 양방향 |
| **프로토콜** | HTTP | WS (별도 프로토콜) |
| **실시간성** | ~100ms | ~50ms |
| **자동 재연결** | **브라우저 내장** | 직접 구현 필요 |
| **방화벽/프록시** | **HTTP라 문제 없음** | WS 차단하는 프록시 존재 |
| **바이너리 전송** | 불가 (텍스트만) | 가능 |
| **연결 제한** | HTTP/1.1: 도메인당 6개 (HTTP/2: 해결) | 없음 |

## 이중화 비교 (핵심)

### 아키텍처 차이

```
[SSE + REST 이중화]

Client ─ POST(전송) ─→ LB(라운드로빈) ─→ Server 1 or 2
Client ─ SSE(수신)  ─→ LB             ─→ Server 1 (끊기면 자동 재연결)
                                            │
Server 1 ──── Redis Pub/Sub ──── Server 2

→ 전송: stateless (아무 서버에 분배)
→ 수신: stateful이지만 자동 재연결
→ LB: 단순 라운드로빈 OK
```

```
[WebSocket 이중화]

Client ─ WS(전송+수신) ─→ LB(Sticky Session) ─→ Server 1 (고정)
                                                    │
Server 1 ──── Redis Pub/Sub ──── Server 2

→ 전송+수신 모두: stateful (같은 서버 유지 필수)
→ LB: Sticky Session 필수
```

### 부하 비교

| 항목           | SSE + Redis |   WS + Redis   | WS + STOMP + RabbitMQ |
| ------------ | :---------: | :------------: | :-------------------: |
| 서버 메모리 (연결당) |    ~2KB     |     ~4~8KB     |    ~4~8KB + STOMP     |
| Redis 부하     |   Pub/Sub   |    Pub/Sub     |          없음           |
| 외부 브로커       |     없음      |       없음       |      RabbitMQ 필요      |
| LB 부하        |  낮음 (HTTP)  | 높음 (Sticky+유지) |    높음 (Sticky+유지)     |
| 네트워크 오버헤드    | HTTP 헤더 반복  |   낮음 (프레임만)    |    중간 (STOMP 프레임)     |

### 서버 장애 시

| 상황       | SSE + REST                    | WebSocket                   |
| -------- | ----------------------------- | --------------------------- |
| 서버 1대 죽음 | SSE 자동 재연결 → 다른 서버. 전송은 영향 없음 | 모든 WS 세션 끊김. 재연결+재인증+재구독 필요 |
| 복구 시간    | ~1초 (브라우저 자동)                 | ~3~5초 (구현 의존)               |
| 구현 복잡도   | 없음 (브라우저가 처리)                 | 재연결 로직 + 미수신 메시지 복구 로직 필요   |

### 동시 1,000명 기준 리소스

| 항목        | SSE                  | WebSocket        |
| --------- | -------------------- | ---------------- |
| 서버 메모리    | ~2MB                 | ~8MB             |
| Redis 메시지 | 동일                   | 동일               |
| LB 커넥션    | 1,000 HTTP + REST 요청 | 1,000 WS (영구 유지) |

## 구현 복잡도 비교

| 항목 | SSE + REST | WS + Redis | WS + STOMP + RabbitMQ |
|------|-----------|------------|----------------------|
| 서버 코드 | REST 컨트롤러 + SSE 엔드포인트 | WebSocket 핸들러 + 라우팅 | STOMP 설정 + @MessageMapping |
| 인증 | 기존 Spring Security 그대로 | WebSocket 핸드셰이크 인증 별도 구현 | STOMP ChannelInterceptor 별도 구현 |
| 이중화 | Redis Pub/Sub 추가 | Redis Pub/Sub + 세션 관리 | RabbitMQ 설치 + 릴레이 설정 |
| 클라이언트 | EventSource API + fetch | WebSocket API + 재연결 로직 | SockJS + STOMP.js + 재연결 |
| 별도 앱 필요 | **불필요** (기존 web에 추가 가능) | 별도 app/chat 권장 | 별도 app/chat 필요 |
| 추가 인프라 | 없음 (Redis 기존 사용) | 없음 (Redis 기존 사용) | RabbitMQ 서버 |

## 방식별 코드 예시 (Spring Boot)

### SSE + REST

```java
// 메시지 전송 (일반 REST)
@PostMapping("/api/v1/chat/msg")
public ResponseEntity<Void> sendMessage(@RequestBody ChatMsgCrtRq rq) {
    chatMsgService.sendMessage(rq);  // DB 저장 + Redis Publish
    return ResponseEntity.ok().build();
}

// 메시지 수신 (SSE 스트림)
@GetMapping(value = "/api/v1/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter subscribe() {
    String userNo = SecurityHelper.getAuthUser().getNo();
    SseEmitter emitter = new SseEmitter(Long.MAX_VALUE);
    sseService.addEmitter(userNo, emitter);  // Redis Sub로 메시지 수신 시 emitter.send()
    return emitter;
}
```

### WebSocket + Redis

```java
// WebSocket 핸들러
@Override
public void handleTextMessage(WebSocketSession session, TextMessage message) {
    ChatWsMessageRq rq = JsonUtil.fromJson(message.getPayload(), ChatWsMessageRq.class);
    chatMsgService.sendMessage(rq);  // DB 저장 + Redis Publish
}

// Redis Sub → 로컬 세션에 push
@Override
public void onMessage(Message message) {
    ChatWsMessageRs msg = deserialize(message);
    sessionManager.getSessionsByRoom(msg.getRoomId())
        .forEach(session -> session.sendMessage(new TextMessage(toJson(msg))));
}
```

## 추천

| 우선순위 | 방식 | 이유 |
|---------|------|------|
| **1** | **SSE + REST** | 이중화 가장 간단, 추가 인프라 없음, 장애 복구 자동, 기존 앱에 통합 가능 |
| 2 | WebSocket + Redis | 양방향 필요 시, SSE 대비 약간 낮은 지연 |
| 3 | WS + STOMP + RabbitMQ | 대규모 서비스, RabbitMQ 인프라가 이미 있는 경우 |

> [!note]
> 채팅에서 "전송"은 빈도가 낮고 (사용자가 타이핑하는 시간), "수신"이 주된 트래픽이므로 단방향 SSE로도 충분하다. 50ms vs 100ms 차이는 체감하기 어렵다.
