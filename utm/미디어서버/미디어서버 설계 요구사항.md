# UTM 미디어서버 설계 요구사항

> 작성일: 2025-02-16
> 상태: 요구사항 정리 → 오픈소스 선정 전
> 관련 문서: [[미디어서버 비교 (OvenMediaEngine vs MediaMTX)]]

---

## 1. 프로젝트 배경

- 상용 서비스에 적용할 미디어 서버가 필요
- 자체 구현할 시간이 없어 **오픈소스로 대체**
- 부하 처리(클러스터링)가 가능해야 함

---

## 2. 시스템이 해야 할 일

```
[Client] ──영상 URL 제공──→ ┌─────────────────────┐
                            │    미디어 서버 시스템    │
                            │                     │
                            │  ① 수신 (Ingest)     │ ← Client가 준 URL로 영상 Pull
                            │  ② 저장 (Recording)  │ → 스토리지에 파일 저장
                            │  ③ 실시간 송출 (Live)  │ → 웹 브라우저 시청자
                            │  ④ VOD 송출          │ → 웹 브라우저 시청자 (개인별 제어)
                            │  ⑤ 재전송 (Push)      │ → 다른 서버
                            └─────────────────────┘
```

| # | 기능 | 설명 | 비고 |
|---|------|------|------|
| ① | **영상 수신** | Client가 제공한 URL에서 영상을 가져옴 (Pull 방식) | RTSP/RTMP/SRT 등 |
| ② | **영상 저장** | 수신한 영상을 서버 스토리지에 녹화 | MP4/TS/fMP4 등 |
| ③ | **실시간 송출** | 수신 중인 영상을 실시간으로 시청자에게 전달 | 웹 브라우저 대상 |
| ④ | **VOD 송출** | 저장된 영상을 시청자에게 전달 (빨리감기/되감기/탐색 가능) | 개인별 독립 재생 제어 |
| ⑤ | **타 서버 전송** | 일시적으로 다른 서버에 영상을 Push 전송 | |

---

## 3. 제약 조건

| 항목 | 조건 |
|------|------|
| **용도** | 상용 서비스 |
| **구현 방식** | 오픈소스 활용 (자체 개발 시간 없음) |
| **확장성** | 클러스터링/부하 분산 필수 |
| **시청 환경** | 주로 웹 브라우저 (Chrome, Firefox, Safari) |
| **라이선스** | 상용 서비스에 사용 가능해야 함 |

---

## 4. 아키텍처 분석

### 4.1 Live와 VOD는 별도 구성이 필요

조사 결과, 대부분의 오픈소스 미디어 서버(OvenMediaEngine, MediaMTX)는 **라이브 전용**이다. VOD(저장 영상 개인별 재생)를 위해서는 별도의 HTTP 서버가 필요하다.

```
┌─────────────── 미디어 서버 시스템 ───────────────┐
│                                                │
│  [수신/라이브/녹화]         [VOD 서빙]             │
│  OvenMediaEngine          Nginx (HTTP)         │
│  or MediaMTX              + hls.js (Player)    │
│       │                        ↑               │
│       ├── 실시간 → 웹 브라우저    │               │
│       ├── 녹화 → 스토리지 ───────┘               │
│       └── Push → 다른 서버                       │
│                                                │
└────────────────────────────────────────────────┘
```

### 4.2 재생 모드별 특성

| 모드 | 빨리감기 | 되감기 | 일시정지 | 개인별 독립 | 담당 |
|------|:---:|:---:|:---:|:---:|------|
| Live (실시간) | X | X | X | X | 미디어 서버 |
| VOD (저장 영상) | O | O | O | O | HTTP 서버 + 플레이어 |
| DVR (하이브리드) | 제한적 | O (버퍼 내) | O | O | 미디어 서버 (HLS 세그먼트) |

### 4.3 브라우저 코덱 호환

웹 브라우저가 주 시청 환경이므로 **출력 코덱 선택이 중요**.

| 코덱 | Chrome | Firefox | Safari | WebRTC | 판정 |
|------|:---:|:---:|:---:|:---:|------|
| **H.264** | O | O | O | O | **가장 안전** |
| H.265 | O (MSE) | X | O (MSE) | X | WebRTC 불가 |
| VP8 | O | O | O | O | 안전 |
| AV1 | O | O | O (17+) | O (최신) | 아직 이름 |

> [!important]
> 입력이 H.265일 경우 **트랜스코딩(H.264 변환)이 필수**. 이는 미디어 서버 선정에 직접적으로 영향을 준다.

## 5. 오픈소스 후보 평가

### 5.1 선정 기준 체크리스트

| # | 기준 | OME | MediaMTX | MediaMTX+FFmpeg |
|---|------|:---:|:---:|:---:|
| 1 | 영상 수신 (RTSP/RTMP/SRT) | O | O | O |
| 2 | 녹화/저장 | O | O | O |
| 3 | 실시간 스트리밍 (WebRTC/HLS) | O | O | O |
| 4 | VOD 재생 | **X** (별도 필요) | **X** (별도 필요) | **X** (별도 필요) |
| 5 | 타 서버 Push 전송 | O | O | O |
| 6 | **클러스터링** | **O** (Origin-Edge) | **X** | **X** |
| 7 | **트랜스코딩** | **O** (내장+GPU) | **X** | O (CPU 부하 큼) |
| 8 | ABR (적응형 비트레이트) | O | X | X |
| 9 | 웹 브라우저 호환 출력 | O | O (코덱 맞을 때만) | O |
| 10 | 라이선스 (상용 가능) | GPLv2 (주의) | MIT (자유) | MIT + GPL |
| 11 | 프로덕션 레퍼런스 | O | O (중소규모) | - |

### 5.2 후보별 요약

**OvenMediaEngine (OME)**
- 장점: 트랜스코딩, GPU 가속, ABR, 클러스터링 모두 내장. 상용 수준 기능.
- 단점: GPLv2 라이선스 (서비스 배포 시 확인 필요), 설정 복잡도 중간
- 판정: **요구사항 대부분 충족. 클러스터링이 필수이므로 가장 유력.**

**MediaMTX**
- 장점: MIT 라이선스, 초경량, 설정 간편, 패스스루 성능 우수
- 단점: 트랜스코딩 없음, 클러스터링 없음, VOD 없음
- 판정: **클러스터링 미지원으로 상용 서비스 메인으로는 부적합**

**MediaMTX + FFmpeg 조합**
- 장점: 트랜스코딩 해결, 유연한 구성
- 단점: 채널당 FFmpeg 프로세스 필요 (CPU 부하), 클러스터링 여전히 없음, 수동 관리
- 판정: **프로토타입이나 소규모에는 적합하나 상용 스케일에는 부적합**

### 5.3 잠정 결론

```
Live/녹화 서버:  OvenMediaEngine (클러스터링 + 트랜스코딩)
VOD 서버:       Nginx + hls.js (별도 구성 필요)
보조 도구:       FFmpeg (필요 시 추가 변환)
```

## 6. 구성 예상 아키텍처

```
                         ┌──────────────────────────────────────┐
                         │         OvenMediaEngine Cluster       │
                         │                                      │
[Client URL] ──Pull──→  Origin 서버                              │
                         │  - 수신 (RTSP/RTMP/SRT)              │
                         │  - 트랜스코딩 (H.265→H.264)           │
                         │  - ABR (다중 해상도)                   │
                         │  - 녹화 → 스토리지                     │
                         │  - Push → 다른 서버                   │
                         │        │                             │
                         │   Edge 서버 1 ── 시청자 (WebRTC/HLS)  │
                         │   Edge 서버 2 ── 시청자 (WebRTC/HLS)  │
                         │   Edge 서버 N ── ...                  │
                         └──────────────────────────────────────┘
                                    │
                              녹화 파일 저장
                                    ↓
                         ┌─────────────────────┐
                         │    VOD 서빙 (Nginx)   │
                         │    + hls.js Player   │
                         │                     │
                         │  시청자 개인별 제어:     │
                         │  빨리감기/되감기/탐색    │
                         └─────────────────────┘
```

## 7. 미결 사항 (Open Questions)

> 아래 항목의 답변에 따라 아키텍처와 오픈소스 선정이 달라질 수 있음

| # | 질문 | 영향 |
|---|------|------|
| 1 | 입력 영상의 코덱/해상도 범위는? (H.264 only? H.265 포함?) | 트랜스코딩 필요 여부 결정 |
| 2 | 동시 수신 채널 수 예상치는? | Origin 서버 사양 결정 |
| 3 | 동시 시청자 수 예상치는? | Edge 서버 수량 결정 |
| 4 | VOD 저장 보존 기간은? | 스토리지 용량 설계 |
| 5 | 인증/접근제어 요구사항은? | Webhook/SignedPolicy 설정 |
| 6 | 온프레미스 vs 클라우드? | 배포 방식, 클러스터링 구성 |
| 7 | 타 서버 전송 시 프로토콜은? (RTMP? SRT?) | Push 설정 |
| 8 | GPLv2 라이선스가 상용 서비스에 문제 없는가? | OME 사용 가능 여부 |
| 9 | DVR(라이브 중 되감기) 기능이 필요한가? | HLS 세그먼트 유지 설정 |
| 10 | 트랜스코딩 시 GPU 서버 확보 가능한가? | CPU only vs GPU 가속 |

## 8. 용어 참조

| 용어 | 설명 |
|------|------|
| **트랜스코딩** | 영상의 코덱/해상도/비트레이트를 변환. H.265 입력 → H.264 출력 등 |
| **GPU 가속** | 트랜스코딩을 GPU(NVENC)에서 처리. CPU 대비 수십 배 효율 |
| **ABR** | 시청자 네트워크에 따라 화질 자동 전환 (1080p↔720p↔480p) |
| **클러스터링** | Origin-Edge 구조로 서버 수평 확장. Edge 추가로 시청자 수 확장 |
| **패스스루** | 코덱 변환 없이 그대로 전달. 가볍지만 코덱 불일치 시 재생 불가 |
| **VOD** | Video on Demand. 저장된 영상을 개인별 독립 재생 (빨리감기 등) |
| **DVR/Timeshift** | 라이브 중에도 과거 버퍼 범위 내 되감기 가능한 하이브리드 모드 |
| **FFmpeg** | 오픈소스 영상/음성 변환 도구. MediaMTX에 트랜스코딩 기능을 보완 |
| **HLS** | HTTP Live Streaming. Apple 개발, 브라우저 호환성 높음 |
| **WebRTC** | 초저지연(~0.5s) 실시간 스트리밍 프로토콜 |
| **LL-HLS** | Low-Latency HLS. 일반 HLS(5-30s)보다 지연 낮음(1-4s) |
